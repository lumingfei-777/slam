前后端数据流 Frontend → Backend · 前端产出什么，后端消费什么

用一张图 + 代码说明：前端每帧产生的数据如何组织和传递给后端。

一、一句话回答

不需要"单独写程序保存"——数据就在内存里，用共享对象（Map）传递。

前端往 Map 里写关键帧和 3D 点，后端从 Map 里读它们做优化。
加线程锁保护即可，不需要序列化到磁盘。

二、前后端之间的完整数据流

前端 每帧

│ 输入：左右图像
│ 输出：当前帧位姿 $T_k$ + 新 3D 点 ${P_j}$ + 关联 ${kp \leftrightarrow point}$
│
├── 普通帧 → 位姿存到 trajectory[]（用于输出，不用于后端）
│
└── 关键帧 → 写入 map.keyframes[] + map.points{}
    （这就是传给后端的数据）

后端 处理关键帧时

│ 从 map.keyframes[] 读最近 N 个 KF
│ 从 map.points{} 读它们观测到的 3D 点
│ 执行 BA → 更新 KF 的位姿和点的坐标
│ 直接修改 Map 里的数据（不需要传回）

三、核心数据结构：Map 类

Map 是前后端之间的共享存储。前端往里写，后端从里读：

class Map:
  # ── 前端写入，后端读取 ──
  # 1. 所有关键帧
  keyframes: list[KeyFrame]   # ← 前端追加，后端读取 + 修改

  # 2. 所有地图点
  points: dict{int: MapPoint}  # ← 前端添加，后端读取 + 修改坐标

关键帧里存什么（KeyFrame）：

class KeyFrame:
  pose: np.ndarray (4,4)    # ← 后端 BA 会修改这个
  pts_2d: list[(float,float)]    # 观测到的 2D 特征点位置（只读）
  point_ids: list[int]      # 对应的地图点 ID → 用于 BA 构建观测方程
  timestamp: int        # 时间戳（排序用）

地图点里存什么（MapPoint）：

class MapPoint:
  position: np.ndarray (3,)    # ← 后端 BA 会修改这个
  descriptor: np.ndarray (32,)   # ORB 描述子（用于匹配，只读）
  id: int          # 全局唯一 ID

四、前端和后端各自读写哪些数据

操作谁做的读什么写/改什么
添加新关键帧 前端 当前帧位姿、特征点 map.keyframes.append()
添加新 3D 点 前端 三角化得到的 3D 坐标 map.points[id] = pt
更新特征-3D点关联 前端 匹配结果 m.queryIdx ↔ m.trainIdx 自身的 prev_pt_map（下一帧用）
读取最近 KF 做局部 BA 后端 map.keyframes[-N:] 修改 KF.pose
读取 3D 点做 BA 后端 map.points[id] 修改 point.position
闭环检测 后端 所有 KF 的位姿 + 描述子 全局修改 KF.pose
⚠️ 核心要点：前后端共享的是同一份数据。

Map 本身就是一个"写程序保存"的存储空间——它就在内存里。
多线程时加锁：
with map.lock:
    map.keyframes.append(new_kf)

第 k 帧

│
前端 立体匹配 → 三角化 → 新 3D 点 ${P_j}$
│
前端 写 map.add_point(P_j)  （Map 中多了 N 个点）
│
前端 判断：是不是关键帧？ → 否 → 结束
│           → 是，继续：
│
前端 写 map.add_keyframe($T_k$, pts_2d, point_ids)
│
后端 读 map.keyframes[-3:] → 最近 3 个 KF
后端 读 map.points[id] → 它们共同看到的 3D 点

│

前端 读 map.points{} → 用更新后的 3D 点做 PnP
│

七、常见疑问

Q1：前后端如果分线程，数据不会冲突吗？

会的。所以需要互斥锁（Lock）保护共享数据：

import threading
class Map:
    def __init__(self):
        self.lock = threading.Lock()
        self.keyframes = []
        self.points = {}
    
    def add_keyframe(self, kf):
        with self.lock:
            self.keyframes.append(kf)
    
    def get_last_kfs(self, n):
        with self.lock:
            return self.keyframes[-n:]

Q2：后端修改了 KF 位姿后，前端的位姿会变吗？

不会直接变。前端用的是自己维护的 self.prev_pose，不是从 Map 读的。
但下一帧做 PnP 时，用的 3D 点是 Map 里的——如果后端修改了它们的坐标，PnP 的结果就会受后端的优化影响。

Q3：所有帧的位姿都要存吗？

所有帧的位姿存在 self.all_poses 里（只用于输出轨迹）。
只有关键帧的位姿存在 Map 里（用于后端优化）。

Q4：后端什么时候触发？

两种方式：

  • 同步触发（你目前的代码）：处理完一帧后立即调用 local_bundle_adjustment()
  • 异步触发（生产环境）：后端线程不断检查队列，有新的 KF 就处理

八、总结

前端（每帧）

 特征提取 → 匹配 → PnP → 位姿
 立体匹配 → 三角化 → map.add_point()
 关键帧判断 → 是 → map.add_keyframe()

后端（关键帧到达时）

 从 map 读最近 KF + 点的数据
 执行 BA → 修改 map 中的 KF 位姿和点坐标
 下一帧 PnP 时自动用更新后的点

Map 是唯一需要"保存"的数据结构——它就在内存里，前后端共享
